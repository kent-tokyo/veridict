# リサーチマップ

[English](research-map.md) | 日本語

検討したものの実装していない(または実装しない)候補と、その理由、そして判断が変わりうる条件を
まとめます。これは「保留であり却下ではない」リストであり、ロードマップではありません - 今後これらの
どれかについて要望があれば、それは正当な再検討であり、一度決着した問題の蒸し返しではありません。
実装済みの内容とその統計的根拠は `docs/metrics_ja.md` を参照してください。

## 検討したが未実装の統計手法

### Wilson連続性補正(`winrate`/`sign-test`/`elo`)

**内容:** Wilson score intervalに対する小サンプル向けの補正で、区間をわずかに広げることで、
カバレッジの正確性のために保守性の一部を犠牲にするものです。

**未実装の理由:** これは以前の「精度改善」という優先事項の下で始まった2つの精度調査のうちの1つ
です。どちらも使える設計を返す前に一時的なツールのエラーで停止し、その後リポジトリの成長/発見性の
改善という別の優先事項に置き換わりました。BCaブートストラップ(もう1つの停止していた調査、こちらは
`mean-diff` と `matrix` の両方で既に実装済み)とは異なり、これは再開する前に本当に注意が必要です:
`wilson_ci` は3つのメトリクス(`winrate`/`sign-test`/`elo`)で共有されているため、正確な連続性補正
の式を権威ある情報源に照らして検証する必要があります - ここの係数が間違っていると、1つのメトリクス
だけでなく、その上に構築されるすべての信頼区間が黙って壊れてしまいます。

**判断が変わる条件:** 小サンプルでの具体的な精度に関する指摘があった場合、あるいは誰かが事前に式の
検証作業を行った場合。

### Normalized-Elo pentanomialとSiegmundの離散時間補正

**内容:** `--sprt-variant pentanomial`(実装済み)は、Fishtestの*ロジスティックElo*版
`LLR_logistic`(期待値制約付き多項MLE)を移植したものです。Fishtestの新しいデフォルトは
*normalized-Elo*(`LLR_normalized`、独自の反復計算を伴う`t`値制約付きMLEで、テストの想定所要
局数が引き分け率や開始局面集合に依存しないようスケールされたもの)です - これは実在する改良ですが、
統計的に妥当なpentanomial検定を得るために必須ではなく、単に(そしてやや複雑な)別のパラメータ化に
過ぎません。別件として、Fishtestは基本のLLRの上にSiegmundの離散時間補正を適用しています - これは
連続的にではなく離散的な間隔でしかチェックしない逐次検定に対して境界をわずかに引き締めるものです。
`trinomial`もこれを適用していないため、`pentanomial`も適用しないことで既存の前例に合わせています -
見落としではありません。

**未実装の理由:** どちらも、既に実装済みで妥当な検定に対する漸進的な精度改善であり、正しさの修正
ではありません - 実装済みのpentanomialが実際に判定基準とする性質(引き分けの多さではなく、ペア内の
相関)については `docs/metrics_ja.md` の `sprt` セクションを参照してください。

**判断が変わる条件:** ロジスティックEloのpentanomialの境界が、引き分け率や開始局面集合によって実務上
無視できないほど変動するという具体的な報告があった場合 - これはまさにnormalized Eloが解決しようと
している問題です。

### 検出力分析 / 必要サンプルサイズのサブコマンド

**内容:** 望ましい効果量と信頼水準が与えられたとき、実験を実行する*前に*何件のトライアルが必要かを
計算するもの(`estimated_additional_trials`が非決定的な結果の後に事後的に行っていることの逆)。

**一部実装済み:** `veridict plan`(`docs/metrics_ja.md`の`plan`セクション参照)が、`matrix`/
トーナメント比較のケースについてこれを行います - `--min-elo`(検出したい効果量)が与えられると、
ペアごとの追加トライアル数を不確実性の高い順に推定します。まだ欠けているもの: 単純な2者間の
`compare`/`sprt`実行(マトリクスを伴わない)についての同じ問い(「どれかを実行する前に、
`--min-effect`のギャップを検出するには何トライアル必要か」)、および固定のトライアル予算の下での
制約付き割り当て(下記のエントリ参照) - こちらは`plan`が意図的に手を付けていません。

### `plan`向けの予算制約付きマッチ割り当て

**内容:** 固定の総トライアル予算(`--budget N`)や明示的なゴール(`--goal identify-best`)が
与えられたとき、実行すべき*最適な*対戦の集合を推薦するもの - すべてのペアを独立に不確実性で
ランク付けする(現在の`plan`が行っていること)のではなく、真の制約付き割り当て/最適化問題です。

**未実装の理由:** どちらについても、このコードベースにも依存先にも実アルゴリズムが存在しません。
`plan`の初期の、より広い設計にはこの両方のフラグが含まれていましたが、実装前にどちらも削られました
- 具体的な要望の裏付けのない投機的な最適化器を構築することは、このプロジェクトが避けている
過剰設計そのものだからです(`AGENTS.md`参照)。`plan`が実際に出荷した`--min-elo`によるランク付き
リストは、これがなくても「次に何をテストすべきか」という動機となったユースケースを既に満たして
います。

**判断が変わる条件:** 各ペアを独立にランク付けする現在の`plan`が、真の同時最適化器より明らかに劣る
推薦を生成する具体的なワークフローがあり、かつ「最適」がここで何を意味すべきかを形作るのに十分な
詳細がある場合(全ペアを目標信頼度に到達させる総トライアル数の最小化か? 単一の最良候補を最速で
特定することか? それとも他の何かか?)。

### 複数メトリクス実行時の多重比較補正

**内容:** `compare`の実行で複数の`--metric`フラグを同時に指定した場合、それぞれが独立に、指定した
信頼水準で判定されます。補正(例: Bonferroni/Holm)なしに複数の検定を実行すると、結果全体での偽陽性
率が増大します。

**未実装の理由:** 検出力分析と同様、実際に価値のあるアイデアではあるものの、まだ具体的な設計に
なっていません。また、`verdict::aggregate`の既存のロジック(「いずれかがfailなら全体がfail」)との
相互作用も検討が必要です(補正は個々の判定が組み合わさる方法ではなく、*どの*個々の判定が有意である
かを変えるものです)。

### `matrix`向けの`--failure-policy`

**内容:** `compare`/`sprt`は`--failure-policy report-only`/`exclude`/`loss`をサポートしています
(`docs/metrics_ja.md`参照)が、`matrix`はサポートしていません - 各候補の失敗は引き続き候補ごとに
タリー・報告されますが、常に`report-only`として振る舞います。

**未実装の理由:** `matrix`には失敗ポリシーが影響を与えるべき判定(verdict)が存在しません
(README の「比較マトリクス」セクションにあるとおり、設計上レポート専用です) - `loss`がしきい値/LLR
境界のどちら側に判定が落ちるかを変える`compare`/`sprt`とは異なります。候補自身のクラッシュ/タイム
アウト率がマトリクス実行における自身のElo推定値を引き下げるべきだという具体的な要望はまだありません。

**判断が変わる条件:** マトリクス実行で、候補のクラッシュ/タイムアウト率を単に報告するだけでなく、
その候補自身のElo推定値を引き下げるべきだという具体的な要望が出てきた場合。

## 意図的にスコープ外としているもの

veridictは結果を判定するものであり、結果を生成するものではありません。これらは「まだ実装していない」
項目ではなく、依存の方向を一方向(統合ツールがveridictに依存し、veridictはそれらに依存しない)に
保つため、意図的に別プロジェクトに委ねているものです:

- Webダッシュボード
- 実験データベース
- クラウドサービス
- 分散ワーカー
- ドメイン固有の対戦実行ツール(例: チェスエンジンやLLMを実行して結果を生成すること)
- LLM judgeの実装
- OCRメトリクスの実装
- 将棋/チェスプロトコルのサポート
- グラフ描画中心のUI

もしあなたのワークフローがこれらのいずれかを必要とするなら、それをveridictにJSONL/CSVを流し込む
別ツールとして構築してください - それが意図された統合の接点であり、欠けている機能ではありません。
